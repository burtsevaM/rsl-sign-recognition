# AGENTS

## Назначение

Этот файл задает рабочие правила для AI-агентов, выполняющих изменения в репозитории `rsl-sign-recognition`.

Он применяется ко всем задачам в этом репозитории и дополняет существующие process-документы, но не заменяет их. Перед началом любой задачи агент обязан свериться с:

- `CONTRIBUTING.md`
- `README.md`
- `docs/backlog.md`
- `docs/roadmap.md`
- `.github/PULL_REQUEST_TEMPLATE.md`

Эти файлы являются source of truth. Если формулировка в `AGENTS.md` когда-либо расходится с ними, приоритет остается за перечисленными документами.

## Общий принцип работы

Агент работает только в рамках одной issue или одной явно поставленной задачи.

Обязательные правила:

- не смешивать несколько задач в одной ветке, одном diff или одном PR;
- не расширять scope "по пути", даже если соседняя правка кажется очевидной;
- не вносить дополнительные улучшения вне acceptance criteria текущей задачи;
- не затрагивать unrelated code, docs, configs и process assets;
- не имитировать выполнение задачи частичным решением, косметическим изменением, фиктивной интеграцией или обходным способом, если acceptance criteria требуют большего;
- если задача требует синхронного обновления документации, контракта или process-описаний, делать это в том же PR.

## Scope и границы clean repo

`rsl-sign-recognition` является clean repository для воспроизводимого ML runtime-контура, а не research sandbox.

Для этого репозитория действуют следующие границы:

- на foundation- и ранних этапах сюда нельзя случайно переносить product code, artifacts, configs, operational scripts, transitional logic и experimental fragments;
- clean repo не должен создавать ложное впечатление, что здесь уже есть полный рабочий ML runtime;
- `gesture-recognition-draft` используется только как источник архитектурной правды, validation context и исходных формулировок;
- перенос из `gesture-recognition-draft` допускается только через отдельные migration tasks;
- без отдельной migration task запрещен перенос кода, артефактов, конфигов, scripts и крупных директорий;
- если задача опирается на материалы draft repo, в scope должны быть явно отделены source context, target scope и non-goals.

## Обязательные файлы для чтения перед началом любой задачи

- `CONTRIBUTING.md` — правила веток, коммитов, issue/PR workflow, требования к синхронному обновлению docs и ограничения на миграцию из draft repo.
- `README.md` — назначение репозитория, текущая стадия clean repo и архитектурные границы между clean repo и draft repo.
- `docs/backlog.md` — канонические task-коды, milestones, epics, зависимости, recommended labels и acceptance criteria.
- `docs/roadmap.md` — порядок milestones, управленческие допущения и принцип "честного состояния" репозитория.
- `.github/PULL_REQUEST_TEMPLATE.md` — обязательная структура PR, checklist и способ привязки PR к issue.

Без чтения этих файлов агент не должен начинать реализацию.

## Ветки

Агент должен работать только в отдельной short-lived branch под одну задачу.

Обязательные правила:

- не работать напрямую в `main`;
- не использовать одну и ту же ветку для нескольких issues;
- имя ветки должно соответствовать типу задачи и конвенции репозитория:
  - `feat/<id>-slug`
  - `fix/<id>-slug`
  - `docs/<id>-slug`
  - `chore/<id>-slug`
- `<id>` должен ссылаться на task code из backlog или на номер issue, если это уместно и task code отсутствует;
- тип ветки должен соответствовать характеру изменения.

## Коммиты

Агент обязан оформлять коммиты по `Conventional Commits`.

Формат:

```text
type(scope): краткое описание
```

Допустимые типы:

- `feat`
- `fix`
- `docs`
- `refactor`
- `test`
- `chore`
- `ci`

Дополнительные правила:

- commit message должен быть осмысленным и конкретным;
- нельзя делать шумные, пустые или дробящие историю коммиты без причины;
- перед коммитом нужно проверить diff и убедиться, что в нем нет случайных несвязанных изменений.

## Pull Request

PR должен соответствовать одной задаче и быть привязан к issue, если issue для этой задачи существует.

Обязательные правила:

- оформлять PR по `.github/PULL_REQUEST_TEMPLATE.md`;
- заполнять все содержательные разделы PR на русском языке;
- допускается использовать английские technical terms, file names, API names, branch names, issue codes и prefixes формата `Conventional Commits`;
- если задача должна закрываться через PR, использовать строку `Closes: #<номер issue>`;
- если отдельной issue нет, не придумывать фиктивную привязку и не указывать ложный номер;
- честно описывать, как изменение проверялось; если проверка неприменима, нужно явно написать почему.

## Issue workflow

Рекомендуемый порядок работы агента:

1. Выбрать одну issue или одну явно поставленную задачу.
2. Проверить milestone, epic, зависимости и acceptance criteria.
3. Создать отдельную ветку по правилам репозитория.
4. Выполнить изменение только в пределах согласованного scope.
5. При необходимости синхронно обновить документацию, контракт или process-описания.
6. Проверить diff на предмет unrelated изменений и случайных элементов из draft repo.
7. Сделать commit по правилам репозитория.
8. Открыть PR по шаблону и привязать его к issue, если issue существует.

## Что запрещено делать агенту

- смешивать несколько issues в одной ветке или одном PR;
- переносить код, артефакты, конфиги, scripts или директории из `gesture-recognition-draft` без отдельной migration task;
- менять milestone, epic, dependencies или acceptance criteria без явной задачи на это;
- править unrelated docs, code, CI, templates или process assets;
- создавать ложное впечатление, что clean repo уже содержит полный рабочий runtime;
- делать массовый перенос директорий или крупные копирования из draft repo;
- выполнять скрытые "улучшения" вне scope текущей issue;
- закрывать задачу формально, если ее acceptance criteria по факту не выполнены;
- маскировать manual steps, ограничения или отсутствие runtime-поверхности под "готовую" реализацию.

## Самопроверка перед завершением

Перед завершением работы агент должен проверить:

- [ ] scope не расширен;
- [ ] изменены только нужные файлы;
- [ ] ветка создана по правильной конвенции;
- [ ] commit оформлен по правилам `Conventional Commits`;
- [ ] PR будет оформлен по шаблону и на русском языке;
- [ ] не затронуты другие issues;
- [ ] не перенесены случайные элементы из `gesture-recognition-draft`;
- [ ] diff не создает ложного впечатления, что clean repo уже содержит полный рабочий ML runtime.
